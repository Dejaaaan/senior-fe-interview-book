Internationalization (the process of preparing an application for delivery in multiple languages and locales) appears on the surface to be the act of swapping strings, but at scale it encompasses plurals, gender, dates, numbers, currencies, right-to-left layouts, and a translation workflow that ships continuously alongside the rest of the application. Senior interviews probe whether the candidate has internalised the parts and the trade-offs of each.

## The native APIs you should know

The browser's `Intl` namespace handles a surprising amount. Reach for it before a library.

```ts
new Intl.NumberFormat("en-US", { style: "currency", currency: "USD" }).format(1234.5);
// "$1,234.50"

new Intl.NumberFormat("de-DE", { style: "currency", currency: "EUR" }).format(1234.5);
// "1.234,50 €"

new Intl.DateTimeFormat("en-GB", { dateStyle: "long" }).format(new Date());
// "4 May 2026"

new Intl.RelativeTimeFormat("en", { numeric: "auto" }).format(-2, "day");
// "2 days ago"

new Intl.ListFormat("en", { style: "long", type: "conjunction" }).format(["A", "B", "C"]);
// "A, B, and C"

new Intl.PluralRules("ar").select(2); // "two"  (Arabic has 6 plural categories)

const collator = new Intl.Collator("sv");
["Ångström", "Apple", "Banana"].sort(collator.compare);
// Locale-aware sorting
```

A library is not required to format dates, numbers, currencies, or to perform locale-aware sorting; the native `Intl` namespace handles all of that. A library *is* required to manage translated strings and variants — that is, the workflow of authoring source strings in one language, sending them to translators, receiving translated files, and selecting the right string at runtime based on the user's locale.

## ICU MessageFormat

The standard for translated strings with placeholders, plurals, and selects. Most i18n libraries support it.

```text
{count, plural,
  =0 {No items}
  one {# item}
  other {# items}
}
```

```text
{gender, select,
  female {She}
  male {He}
  other {They}
} signed in.
```

International Components for Unicode handles the pluralization cases that the English-centric "you have N item(s)" pattern cannot. Russian distinguishes zero, one, few, many, and other forms. Arabic adds a dual form, distinguishing zero, one, two, few, many, and other. Polish uses complex rules involving the count modulo ten and modulo one hundred. The team should not reinvent this logic — the differences are subtle, the failure modes are linguistically embarrassing, and the rules are encoded in the Common Locale Data Repository that the `Intl` Application Programming Interface and ICU consume directly.

## Library landscape

| Library | Best for |
| --- | --- |
| **`next-intl`** | Next.js App Router projects |
| **`react-intl` (FormatJS)** | Mature React ecosystems; ICU first-class |
| **`i18next` + `react-i18next`** | Framework-agnostic; large plugin ecosystem |
| **`@lingui/core`** | Modern, minimal runtime, ICU |

For Next.js App Router projects, `next-intl` is the framing senior candidates typically present as the default — it integrates cleanly with the Server Components model, supports International Components for Unicode message format natively, and handles locale negotiation, URL routing, and message bundling with minimal configuration.

```ts
function formatPrice(amount: number, currency: string, locale: string) {
  return new Intl.NumberFormat(locale, {
    style: "currency",
    currency,
  }).format(amount);
}
```

## A working `next-intl` setup

```ts
// src/i18n/request.ts
import { getRequestConfig } from "next-intl/server";

export default getRequestConfig(async ({ requestLocale }) => {
  const locale = (await requestLocale) ?? "en";
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

```json
// messages/en.json
{
  "Cart": {
    "items": "{count, plural, =0 {No items} one {# item} other {# items}}",
    "total": "Total: {amount, number, ::currency/USD}"
  }
}
```

```tsx
// app/cart/page.tsx
import { useTranslations } from "next-intl";

export default function CartPage() {
  const t = useTranslations("Cart");
  return (
    <>
      <p>{t("items", { count: 3 })}</p>
      <p>{t("total", { amount: 199 })}</p>
    </>
  );
}
```

## Right-to-Left (RTL) support

The major Right-to-Left languages are Arabic, Hebrew, Persian, and Urdu. The minimum work to support them correctly involves four steps. Set `dir="rtl"` on the `<html>` element for those locales so the browser flips the inline axis automatically. Use logical Cascading Style Sheets properties throughout the codebase — `padding-inline-start`, `margin-inline-end`, `text-align: start` — rather than the directional equivalents `padding-left` and `margin-right`; the logical properties resolve to the correct side based on the document direction, so the layout flips automatically when `dir="rtl"` is set. Mirror icons that have a directional meaning (arrows, chevrons, "back" buttons) but not icons that are direction-neutral (a smiley face does not flip). Test typography: Arabic letterforms have taller ascenders and descenders than Latin script, so the line-height that looks good in English may clip the diacritics in Arabic.

```css
.card { padding-inline: 1rem; }       /* RTL-friendly */
.icon--directional[dir="rtl"] { transform: scaleX(-1); }
```

## Translation workflows

Two extremes bracket the design space. Push-to-file is the simplest workflow: developers add keys and English source strings, push them to the repository, and translators submit Pull Requests with the translated files. This works for small teams with a small number of locales but does not scale because translators are not typically comfortable in Git and Pull Request review of localisation files is awkward. A Translation Management System such as Lokalise, Crowdin, or Phrase is the workflow that scales: developers push English source strings, translators work in the Translation Management System's user interface (which provides translation memory, glossaries, and machine-translation suggestions), and Continuous Integration pulls the translated files back into the repository on a schedule. This works for product teams shipping continuously.

Whichever workflow the team picks, three automations are worth their weight. Lint for missing keys in non-English locales — a Continuous Integration check that fails the build when a key is present in English but missing from another supported locale catches regressions before they ship. Apply pseudo-localization in development and preview environments by wrapping every string in markers (`[!! ñöt-trænslätéd !!]`) so untranslated strings are visually obvious without the developer needing to switch locale. Enforce string length budgets in design review — German strings are typically thirty percent longer than English, Arabic strings are typically shorter; a button designed only against English may overflow in German or look empty in Arabic.

## Locale negotiation

```ts
// Server-side: negotiate based on Accept-Language and a cookie override.
import { match } from "@formatjs/intl-localematcher";

function negotiateLocale(request: Request, supported: string[]) {
  const cookie = request.headers.get("cookie");
  const explicit = parseCookie(cookie ?? "")["NEXT_LOCALE"];
  if (explicit && supported.includes(explicit)) return explicit;

  const accept = request.headers.get("accept-language") ?? "";
  const requested = accept.split(",").map((s) => s.split(";")[0].trim());
  return match(requested, supported, "en");
}
```

URL-prefixed locales (`/en/cart`, `/de/cart`) are SEO-friendlier than cookie-only switching. Most i18n libs default to this.

## What to localize beyond strings

Internationalization extends beyond translated strings into every aspect of the user-visible data model. Numbers, dates, and currencies are handled by the native `Intl` Application Programming Interface and require no library. Time zones should be stored as Coordinated Universal Time and formatted in the user's local time zone using the Internet Assigned Numbers Authority zone identifier (`"Europe/Berlin"`) rather than a Greenwich Mean Time offset, because the offset depends on daylight-saving time and changes through the year. Number and decimal separators differ across locales: French uses spaces as thousands separators (`1 234,56`), Swiss German uses apostrophes (`1'234.56`), and the team's number parser must handle both. Phone numbers should be parsed and formatted with `libphonenumber-js`, which encodes the per-country rules. Addresses have country-specific field orders and required fields — Japan puts the postal code first, Germany puts the house number after the street name, the United States expects "City, State, ZIP" — and the team should not hard-code the United States layout. Names should not assume a two-part "first last" structure; some cultures use a single name, some put the family name first, and the team should design forms with a single "Full name" field unless the structure is needed for a specific reason.

## Performance

Three performance interventions matter. Do not ship every locale's strings to every user; code-split the message bundles per locale and load only the user's locale on initial render. Negotiate the locale at the edge (in middleware or at the Content Delivery Network) so the correct bundle is served on the first byte rather than after a client-side detection round trip. Inline above-the-fold strings into the server-rendered Hypertext Markup Language so the user does not see a flash of untranslated content while the JavaScript that loads the messages is still parsing.

```ts
const formatRelative = new Intl.RelativeTimeFormat(navigator.language, {
  numeric: "auto",
});

formatRelative.format(-2, "day"); // "2 days ago"
```

## Key takeaways

Reach for the native `Intl` Application Programming Interface for formatting numbers, dates, currencies, and lists; reach for a library for managing translated strings and variants. International Components for Unicode message format handles plurals, gender, and selects across the world's languages, including the cases that English-centric pluralization cannot. For Next.js App Router projects, `next-intl` is the framing senior candidates typically present as the default. Right-to-Left support is real work: logical Cascading Style Sheets properties, mirrored directional icons, length budgets, and per-locale typography testing. Use a Translation Management System for any team shipping continuously; pseudo-localization makes untranslated strings visually obvious without switching locale. Localize more than strings — numbers, dates, time zones, phone numbers, addresses, names — using Internet Assigned Numbers Authority time zones rather than Greenwich Mean Time offsets. Code-split per locale, negotiate at the edge, and inline above-the-fold strings into the server-rendered Hypertext Markup Language.

## Common interview questions

1. Why isn't `"Hello, {name}!"` enough as a translation primitive?
2. What does ICU MessageFormat give you that template literals don't?
3. Walk me through making a React layout work in RTL.
4. Trade-offs of cookie-based versus URL-prefixed locale routing?
5. How do you avoid shipping every language's strings to every user?

## Answers

### 1. Why isn't `"Hello, {name}!"` enough as a translation primitive?

A simple placeholder substitution like `"Hello, {name}!"` handles only the simplest case — the literal substitution of a value into a string — and breaks down on every dimension that real internationalization must address. It cannot express plurality ("you have one item" versus "you have two items" versus "you have many items" all use different forms in many languages). It cannot express gender agreement (the verb form, the article, and the adjective ending all change with the subject's gender in many languages). It cannot express selection (different greetings for morning, afternoon, and evening in some locales). It cannot express formality (the second-person pronoun is different in formal and informal address in many languages, and choosing the wrong form is a social error).

**How it works.** International Components for Unicode message format extends the primitive substitution with a small declarative language for plurals, selects, and nested combinations. The translator sees the full structure and can encode the rules of their target language; the runtime resolves the structure based on the variable values. The team writes the source string once in International Components for Unicode format and the translation is itself in International Components for Unicode format, with the language's rules built in.

```text
{count, plural,
  =0 {No items}
  one {# item}
  other {# items}
}
```

**Trade-offs / when this fails.** International Components for Unicode message format adds learning curve for translators who are not familiar with it; the cure is a Translation Management System that exposes the structure as a friendly user interface rather than as raw syntax. The format also cannot express every language's grammar (some languages have agreement rules that depend on multiple variables in ways the format does not directly express), so for very complex cases the team may need to write per-language helpers. The pragmatic answer is that International Components for Unicode handles ninety-five percent of cases and the team should accept the trade-off for the remaining five percent.

### 2. What does ICU MessageFormat give you that template literals don't?

International Components for Unicode message format gives the team three properties that template literals cannot. First, plurality with the per-language rules encoded by the Common Locale Data Repository — the team writes one source string and each translator's plural variants are honoured. Second, selection (gender, formality, time of day) with explicit per-variant text — the translator chooses the right variant for each value, rather than the team trying to express the variants in code. Third, nested combinations — a string can express plural-of-gender or gender-of-plural in a single expression, which would be unreadable in template-literal form.

**How it works.** A template literal evaluates the expression and concatenates the result; the team has no way to express variation based on the value's properties. International Components for Unicode message format defines a small declarative language with `plural`, `select`, `selectordinal`, and number-format syntax; the runtime parses the structure, evaluates the variables, and selects the right variant.

```text
{count, plural,
  =0 {You have no messages.}
  one {You have one message from {sender}.}
  other {You have # messages from {sender}.}
}
```

**Trade-offs / when this fails.** International Components for Unicode message format requires runtime parsing and evaluation, which is more expensive than a template literal; for a small number of strings on the critical path, the cost is negligible, but for thousands of strings on every render it becomes measurable. The cure is to compile the messages at build time with a tool like `@formatjs/cli` so the runtime cost is amortised. The format also requires a parser library, which adds bundle weight; the cure is to use one of the modern lightweight implementations (the `intl-messageformat` library is approximately 8 kilobytes minified and gzipped).

### 3. Walk me through making a React layout work in RTL.

The work has three layers. At the document level, set `dir="rtl"` on the `<html>` element when the user's locale is a Right-to-Left language; this causes the browser to flip the inline axis automatically. At the styling level, use logical Cascading Style Sheets properties throughout the codebase — `padding-inline-start` instead of `padding-left`, `margin-inline-end` instead of `margin-right`, `text-align: start` instead of `text-align: left` — so the layout flips automatically when the document direction flips. At the asset level, identify icons that have a directional meaning and mirror them per-direction (an arrow pointing right in Left-to-Right should point left in Right-to-Left); icons that are direction-neutral (a smiley face) should not be mirrored.

**How it works.** Logical Cascading Style Sheets properties resolve to physical properties based on the document's writing mode and direction. In Left-to-Right languages, `padding-inline-start` resolves to `padding-left`; in Right-to-Left languages, it resolves to `padding-right`. The browser handles the resolution; the developer writes the styles once. Directional icons need explicit handling because the browser cannot know which icons should flip; the team's design system should mark directional icons and apply a `transform: scaleX(-1)` in Right-to-Left contexts.

```css
.card {
  padding-inline: 1rem;
  margin-inline-start: 0.5rem;
  text-align: start;
}

.icon--directional[dir="rtl"] {
  transform: scaleX(-1);
}
```

**Trade-offs / when this fails.** Logical properties require browser support; modern browsers support them but very old browsers do not, which is rarely a concern in 2026 but worth checking against the team's browser-support matrix. Some third-party components are written with physical properties and will not flip automatically; the cure is to either pick components that support Right-to-Left out of the box (Radix UI, React Aria, MUI) or to add per-component overrides in Right-to-Left mode. Typography is the most subtle layer — Arabic letterforms have taller diacritics than Latin and may clip with line heights tuned for English — and requires per-language testing rather than a single global setting.

### 4. Trade-offs of cookie-based versus URL-prefixed locale routing?

URL-prefixed routing (`/en/cart`, `/de/cart`) is the better default for most applications because it makes the locale explicit in the Uniform Resource Locator, which is essential for Search Engine Optimization (search engines can index each locale separately and present the right one in the right region's results), shareability (a link to the German page actually opens the German page for the recipient), and caching (the Content Delivery Network can cache each locale independently with no cookie-based variance). Cookie-based routing keeps the Uniform Resource Locator clean but requires the server to inspect the cookie on every request, complicates caching, and breaks shareability — sending a Uniform Resource Locator to a user with a different locale cookie shows them the wrong language.

**How it works.** Uniform Resource Locator-prefixed routing places the locale as the first segment of the path; the server matches the locale, loads the corresponding messages, and renders the page. Cookie-based routing places the locale in a cookie set by the server; every request to the application reads the cookie and selects the locale. The hybrid approach uses Uniform Resource Locator prefixing for the canonical Uniform Resource Locator and a cookie for the user's preference, with the cookie used only to redirect the user to the right Uniform Resource Locator on first visit.

```ts
import { match } from "@formatjs/intl-localematcher";

function negotiateLocale(request: Request, supported: string[]) {
  const cookie = request.headers.get("cookie") ?? "";
  const explicit = parseCookie(cookie)["NEXT_LOCALE"];
  if (explicit && supported.includes(explicit)) return explicit;
  const accept = request.headers.get("accept-language") ?? "";
  return match(accept.split(",").map((s) => s.split(";")[0].trim()), supported, "en");
}
```

**Trade-offs / when this fails.** Uniform Resource Locator-prefixed routing requires the team to handle the locale in every internal link, which is operational work; the cure is a typed wrapper around `<Link>` that automatically prefixes the current locale. The pattern also produces multiple Uniform Resource Locators for the same logical resource (one per locale), which the team must mark as alternates with `<link rel="alternate" hreflang="...">` tags so search engines understand the relationship. Cookie-based routing is acceptable for applications where Search Engine Optimization is irrelevant (an internal admin tool, a logged-in dashboard) but should not be used for any user-facing marketing surface.

### 5. How do you avoid shipping every language's strings to every user?

Code-split the message bundles per locale, negotiate the locale at the edge so the team knows which bundle to send before the request reaches the application, and load only the user's locale on initial render. The size impact is substantial — an application that supports twenty locales with thousands of messages can easily ship hundreds of kilobytes of unused translations on every page load if the bundles are not split.

**How it works.** Each locale's messages are stored in a separate file (`messages/en.json`, `messages/de.json`, ...) and loaded on demand using a dynamic import keyed by the resolved locale. Edge middleware reads the user's locale from the Uniform Resource Locator path or the cookie and either rewrites the request to the appropriate locale-prefixed Uniform Resource Locator or sets a header that the application uses to select the bundle. The Content Delivery Network caches each locale's response independently, so users in the same locale share the same cached response.

```ts
// next-intl loads messages per locale on demand.
import { getRequestConfig } from "next-intl/server";

export default getRequestConfig(async ({ requestLocale }) => {
  const locale = (await requestLocale) ?? "en";
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

**Trade-offs / when this fails.** Code-splitting per locale means that switching locale at runtime requires loading a new bundle, which adds a brief delay. For most users this is acceptable because they rarely switch locale; for users who do switch frequently (a multilingual user testing the application), the delay is noticeable. The cure is to preload the alternate locales the user is likely to switch to. The pattern also assumes the locale is known before the page renders; for applications that detect the locale only after a client-side check, the team must either render a skeleton until the messages load or accept a flash of the default locale.
