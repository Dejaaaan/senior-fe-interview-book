Angular's Dependency Injection system is one of the framework's most distinctive features. Coming from React, where consumers typically "just import" the module they need, the framework can look heavy at first encounter — there is a separate vocabulary of providers, tokens, injectors, and hierarchies. The upside is concrete: testability is materially better because every dependency can be substituted at the boundary, lifetimes are explicit so the team controls per-application, per-route, and per-component instances, and the resulting decoupling lets the consumer ask for an interface without knowing which implementation will be wired in at runtime.

> **Acronyms used in this chapter.** API: Application Programming Interface. CD: Change Detection. DI: Dependency Injection. HTTP: HyperText Transfer Protocol. URL: Uniform Resource Locator.

## The simplest case

```ts
import { Injectable, inject } from "@angular/core";
import { HttpClient } from "@angular/common/http";

@Injectable({ providedIn: "root" })
export class TasksService {
  private http = inject(HttpClient);

  list() {
    return this.http.get<Task[]>("/api/tasks");
  }
}
```

```ts
import { Component, inject } from "@angular/core";
import { TasksService } from "./tasks.service";

@Component({ ... })
export class TasksComponent {
  private tasks = inject(TasksService);
}
```

`@Injectable({ providedIn: "root" })` makes `TasksService` a tree-shakeable application singleton. `inject(TasksService)` retrieves it.

## Why DI

The three concrete benefits worth naming in an interview are testability, lifetime control, and decoupling. Testability follows from substitution: in a unit test, the test bed configures the injector to provide a fake `TasksService` instead of the real one, and the component under test runs unchanged. Lifetime control follows from the hierarchy: the team can choose whether a service is a single application-wide instance, a single instance per route, or a fresh instance per component, and the choice is declared by where the provider lives rather than by careful manual instantiation. Decoupling follows from the indirection: the component asks for an `AuthService` token, and the application's bootstrap decides which concrete class implements it — a real implementation in production, a mock in tests, a different real implementation in a feature-flagged branch.

## Providers

A provider tells the injector how to make an instance of a token. Tokens are usually classes, but can be `InjectionToken<T>` for primitives.

```ts
import { InjectionToken } from "@angular/core";

export const API_BASE_URL = new InjectionToken<string>("API_BASE_URL");

bootstrapApplication(AppComponent, {
  providers: [
    { provide: API_BASE_URL, useValue: "https://api.example.com" },
    { provide: TasksService, useClass: ProdTasksService },
  ],
});
```

In tests:

```ts
TestBed.configureTestingModule({
  providers: [
    { provide: API_BASE_URL, useValue: "http://localhost:3000" },
    { provide: TasksService, useClass: FakeTasksService },
  ],
});
```

The four provider recipes have distinct uses. `useClass` instantiates the given class on demand, optionally injecting its constructor parameters; this is the standard form for swapping implementations of a service. `useValue` provides a literal value (a configuration object, a constant, a primitive); this is the standard form for `InjectionToken`-keyed configuration. `useFactory` calls a function that returns the instance, allowing the factory to inject other dependencies and to construct the instance imperatively; this is the form for cases where the construction logic depends on runtime conditions. `useExisting` aliases one token to another, which is useful for backward compatibility when a token is renamed or for exposing the same instance under multiple interfaces.

```ts
// Each recipe expressed as a provider configuration.
providers: [
  { provide: TasksService, useClass: ProdTasksService },           // useClass
  { provide: API_BASE_URL, useValue: "https://api.example.com" }, // useValue
  {
    provide: HttpClientWithAuth,                                  // useFactory
    useFactory: () => new HttpClientWithAuth(inject(AuthService)),
  },
  { provide: LegacyTasksService, useExisting: TasksService },     // useExisting
]
```

## Hierarchical injectors

Each component has its own injector that inherits from its parent. Provide a service at:

- **Root**: one instance for the whole app.
- **Route**: one instance for that route's subtree (using `providers` on the route).
- **Component**: one instance per component instance (using `providers: [...]` on the `@Component`).

```ts
@Component({
  selector: "app-task-list",
  standalone: true,
  providers: [TaskListState],   // new instance per <app-task-list>
  ...
})
export class TaskListComponent {
  private state = inject(TaskListState);
}
```

This is how you get "local state services" — like a hook, but with DI semantics.

## `inject()` vs constructor injection

Pre-Angular-14:

```ts
constructor(private http: HttpClient, private auth: AuthService) {}
```

Modern:

```ts
private http = inject(HttpClient);
private auth = inject(AuthService);
```

`inject()` works:

- Inside `@Injectable` and `@Component` class bodies.
- Inside factory providers.
- Inside route guards / resolvers (function-based; the modern style).

```ts
import { inject } from "@angular/core";
import { CanActivateFn, Router } from "@angular/router";
import { AuthService } from "./auth.service";

export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);
  if (auth.isLoggedIn()) return true;
  router.navigate(["/login"]);
  return false;
};
```

Old class-based guards are deprecated. Function guards are 2026-standard.

## Optional and self / skip-self

```ts
import { inject, Optional, Self, SkipSelf } from "@angular/core";

const logger = inject(Logger, { optional: true }) ?? noopLogger;
```

The three modifiers tune how the injector resolves the token. `optional: true` returns `null` (rather than throwing) when no provider is found, which is the right shape for soft dependencies — log sinks, optional configuration, feature-flag-gated services. `self: true` restricts the lookup to the current component's own injector and does not climb to parents, which is the right shape for assertions like "this directive must be on a component that explicitly provides this token". `skipSelf: true` does the opposite — it skips the current injector and resolves the token starting from the parent, which is the standard pattern for components that compose with their parent (a `<form>` field that locates the nearest parent `FormGroup`, a child component that delegates to the parent's state).

```ts
// A directive that requires a parent ControlContainer and never resolves itself.
@Directive({ selector: "[appField]", standalone: true })
export class FieldDirective {
  private container = inject(ControlContainer, { skipSelf: true });
}
```

These modifiers let the team build hierarchically-aware components — a form field that finds its parent group, a tab panel that finds its parent tab set, a cell that finds its parent table.

## Tree-shaking

`@Injectable({ providedIn: "root" })` services are tree-shakeable: if no consumer imports them, they're dropped from the bundle. `providers: [Service]` in module/component is not tree-shaken — it's eagerly registered.

For app-wide singletons in standalone apps, use `provideSomething()` factory functions:

```ts
bootstrapApplication(App, {
  providers: [
    provideHttpClient(),
    provideRouter(routes),
    provideAnimations(),
  ],
});
```

These compose cleanly and tree-shake well.

## Anti-patterns

Three anti-patterns are common enough to be worth naming. Instantiating a service with `new` instead of injecting it bypasses the DI system entirely and makes the consumer untestable, because there is no boundary at which the test can substitute a fake. Static singletons (a class with `static instance` accessed by importers) suffer the same problem — the test cannot swap the implementation without rewriting the consumer. Providing the same token at multiple levels of the hierarchy without a clear reason creates confusion about which instance any given consumer will receive; the cure is to pick a level (root, route, or component) and document the reason for that choice in the service's source comment.

## Key takeaways

- Dependency Injection is core to Angular. Use `inject()` over constructor injection in all new code, and use function-based guards and resolvers in routing.
- `providedIn: "root"` is the right default for tree-shakeable application singletons; if no consumer imports the service, the build drops it from the bundle.
- Hierarchical injectors enable per-component "local services" by listing the service in the component's `providers` array, which yields one instance per component instance.
- Function-based guards and resolvers via `inject()` are the 2026 standard, replacing the deprecated class-based forms.
- The `useClass`, `useValue`, `useFactory`, and `useExisting` provider recipes cover every wiring need; the most common recipe in tests is `useClass` with a fake implementation.

## Common interview questions

1. What does `providedIn: "root"` do?
2. How would you provide a different `TasksService` in tests?
3. Difference between providing at root vs component level?
4. What's `inject()` and why is it preferred over constructor DI?
5. How do tree-shakeable providers work?

## Answers

### 1. What does `providedIn: "root"` do?

`providedIn: "root"` declares the service as an application-wide singleton registered with the root injector and, crucially, makes the registration tree-shakeable. The framework instantiates the service the first time any consumer requests it, holds the instance for the lifetime of the application, and reuses it for every subsequent request. The tree-shaking property is what distinguishes the root form from listing the service in a `providers` array: if no consumer imports the service, the build excludes it from the bundle entirely.

**How it works.** The compiler attaches the registration to the service class itself (via the `@Injectable` metadata). The application's tree-shaker can therefore prune unused services because the registration travels with the class rather than living in a centralised list that the tree-shaker cannot statically analyse. When a consumer requests the service, the framework's injector instantiates it on demand, injecting the service's own dependencies recursively.

```ts
@Injectable({ providedIn: "root" })
export class TasksService {
  private http = inject(HttpClient);
  list() { return this.http.get<Task[]>("/api/tasks"); }
}
// Used anywhere via inject(TasksService); one instance per application.
```

**Trade-offs / when this fails.** The root form is the right default for stateless services and for stateful services whose state is shared across the application (an authentication service, a user-preferences cache). It is the wrong choice for services that hold per-route or per-component state — a form's draft state, a wizard's progress — because every consumer would share the same instance and state would leak between unrelated views. The cure for those cases is to provide the service at the route or component level, accepting that the registration is no longer tree-shakeable in exchange for the correct lifetime.

### 2. How would you provide a different `TasksService` in tests?

The test bed configures the injector to provide a fake implementation in place of the real one, using one of the provider recipes (`useClass`, `useValue`, `useFactory`). The component under test is unchanged — it requests `TasksService` via `inject(TasksService)` exactly as in production — and receives the fake because the test bed's provider takes precedence over the root registration.

**How it works.** The test bed creates an injector for the test, populates it with the test's providers, and any `inject()` call inside a component instantiated by the test bed resolves through that injector. The provider override does not affect the production code at all; it is purely a test-time substitution at the injector boundary.

```ts
import { TestBed } from "@angular/core/testing";
import { TasksComponent } from "./tasks.component";
import { TasksService } from "./tasks.service";

class FakeTasksService {
  list = () => of([{ id: "1", title: "test", done: false }]);
}

beforeEach(() => {
  TestBed.configureTestingModule({
    imports: [TasksComponent],
    providers: [{ provide: TasksService, useClass: FakeTasksService }],
  });
});

it("renders tasks", () => {
  const fixture = TestBed.createComponent(TasksComponent);
  fixture.detectChanges();
  expect(fixture.nativeElement.textContent).toContain("test");
});
```

**Trade-offs / when this fails.** The pattern requires the consumer to inject the service rather than instantiate it directly with `new`, which is the senior reason to avoid `new` for application services. The pattern also requires the fake to match the real service's interface, which is best enforced by extracting an interface (or by using `Pick<TasksService, "list" | "create">` for partial doubles). The pattern fails when the test substitutes too narrow a fake and the component calls a method the fake does not implement; the cure is to have the fake throw an explicit error for unimplemented methods rather than silently returning `undefined`.

### 3. Difference between providing at root vs component level?

Providing at root produces a single application-wide instance, shared by every consumer for the lifetime of the application. Providing at a component level produces one instance per component instance — every time the component is mounted, a fresh service is created; every time it is unmounted, the service is disposed. The choice expresses the lifetime the team intends for the service's state.

**How it works.** Each component has its own injector, which inherits from its parent's injector. When a consumer requests a token, the injector first checks itself, then climbs the parent chain until it finds a provider or hits the root. Providing the token at the component level installs a provider on the component's own injector, which short-circuits the climb and gives every consumer beneath the component the component-scoped instance.

```ts
@Component({
  selector: "app-task-list",
  standalone: true,
  providers: [TaskListState],     // new instance per <app-task-list>
  template: `<!-- ... -->`,
})
export class TaskListComponent {
  private state = inject(TaskListState);   // resolved from this component's own injector
}
```

**Trade-offs / when this fails.** Component-level providers are the right shape for "local state services" — the equivalent of a custom hook in React, but with DI semantics. They are the wrong choice for services with expensive initialisation (a long-lived WebSocket, a heavy in-memory cache) because the framework will discard the instance every time the component unmounts. The cure for those is to keep the heavy state at root and to keep the per-component view-model service light. Providing at multiple levels by accident — root and component, route and component — produces confusion about which instance any consumer receives; the cure is a deliberate documentation comment on the service explaining the intended lifetime.

### 4. What's `inject()` and why is it preferred over constructor DI?

`inject()` is the function-form alternative to constructor injection introduced in Angular 14. The consumer calls `inject(SomeToken)` from inside a class field initialiser, an `@Injectable` factory, a provider factory, a function-based guard, or a function-based resolver, and receives the token's instance. The function form is preferred over constructor injection because it has materially better TypeScript inference, can be called from helper functions outside the class body, and removes the constructor noise that grows linearly with the number of dependencies.

**How it works.** `inject()` reads from the framework's "current injector context", which is set when the framework is in the middle of constructing or invoking an injectable scope. The function works inside class field initialisers because Angular sets the injector context before calling the constructor, and works inside guards and resolvers because the router sets the injector context before invoking them.

```ts
// Old: constructor injection — verbose and noisy with many deps.
@Injectable({ providedIn: "root" })
export class OldService {
  constructor(
    private http: HttpClient,
    private auth: AuthService,
    private logger: Logger,
  ) {}
}

// New: inject() — concise, with normal class field syntax.
@Injectable({ providedIn: "root" })
export class NewService {
  private http = inject(HttpClient);
  private auth = inject(AuthService);
  private logger = inject(Logger);
}
```

**Trade-offs / when this fails.** `inject()` only works inside an injector context; calling it from a regular function or a component event handler outside the construction phase throws an error. The cure is to call `inject()` in field initialisers (or in functions called during construction) and to capture the result in a field for later use. The function form also makes it harder to spot, in code review, that a class has many dependencies — the constructor signature was previously a quick visual signal of dependency count. The cure is the team convention of grouping all `inject()` calls at the top of the class body so the dependency surface is visible at a glance.

### 5. How do tree-shakeable providers work?

A tree-shakeable provider is one whose registration is attached to the service class itself rather than to a centralised provider list. When the build's tree-shaker analyses the dependency graph, it can prune the service if no consumer imports it, because the registration is a property of the (now-unreachable) class rather than a separate import the tree-shaker cannot statically prove is unused. This is the property `providedIn: "root"` provides: the registration travels with the class, so unused services are dropped from the bundle.

**How it works.** The compiler emits an `ɵprov` (provider) static property on the service class, which the framework's injector reads at lookup time. The build's tree-shaker can analyse the import graph and determine which classes are reachable from the application's entry points; classes with no inbound import edge are pruned, and their `ɵprov` properties go with them. The legacy form — listing the service in a module's or component's `providers` array — is not tree-shakeable because the array forces the service to be imported by the module that lists it, even if no consumer uses the service.

```ts
// Tree-shakeable: the registration is on the class.
@Injectable({ providedIn: "root" })
export class AnalyticsService {
  track(event: string) { /* ... */ }
}

// Not tree-shakeable: the providers array forces the import.
@NgModule({ providers: [AnalyticsService] })
export class CoreModule {}
```

**Trade-offs / when this fails.** Tree-shakeable providers are nearly always the right choice; the modern `provideX()` factory functions (`provideHttpClient`, `provideRouter`, `provideAnimations`) follow the same model and compose cleanly in the application's `bootstrapApplication` call. The pattern fails when the service has side effects on instantiation that the application depends on regardless of whether any consumer reads it (an analytics initialiser, a global error handler); for those, list the service in the bootstrap providers explicitly so it is instantiated even if no consumer injects it.
